# Migration audit — 2026-05-09

End-of-migration sweep. Scope: confirm parity between the legacy React+Firebase implementation (deleted at commit `f3d115a`, 2026-05-01) and the current SvelteKit+Supabase implementation; surface anything that drifted; record open quality gaps.

**Headline:** all user-facing legacy features are present in the SvelteKit app. Two real gaps were found (one parity, one operational) and a handful of low-severity quality items were collected from the Supabase advisor.

## Method

1. Reconstructed the legacy app from `git show f3d115a^:<path>` — full file inventory, services, hooks, Cloud Functions, Firestore rules, `pl.json`.
2. Inventoried the current SvelteKit app — routes, services, components, types, build config, GH Actions.
3. Inventoried Supabase — migration files vs `mcp__supabase__list_tables` / `list_migrations`. Pulled `get_advisors` (security + performance).
4. Cross-checked feature-by-feature against `MIGRATION_PLAN.md` and the phase tracker in `CLAUDE.md`.

## Parity findings

### Carried over (no gaps)

- **Auth** — Google OAuth, role-based admin gating. Migrated from Firebase custom claims to `app_metadata.role` synced by `sync-user-role` Edge Function.
- **Transactions** — CRUD, recurring templates, multi-month date range filter, category filter, shared-via-group visibility, status lifecycle (`draft|upcoming|overdue|paid`), CSV import/export, bulk delete, transaction detail sheet, click-through category breakdown.
- **Shopping lists** — CRUD, items, completion-to-transaction flow, group sharing, duplicate, rename, item suggestions from history, offline indicator.
- **Categories** — system + user-owned, type-filtered combobox, group-shared visibility.
- **Groups & invitations** — create, invite by email, accept/reject/cancel, transfer ownership, leave, disband, remove-member.
- **Notifications** — in-app bell with unread tracking, weekly admin summary, group-invitation push, role-change notification.
- **Admin pages** — user list with search and role toggle, notifications diagnostic page (token list, manual triggers).
- **PWA** — service worker, installable manifest, dark mode.
- **i18n** — Polish-only; migrated from i18next to Paraglide v2 (compile-time, type-safe).
- **Push** — switched from FCM to native VAPID web-push. Users re-subscribe on first SvelteKit login (intentional clean-cutover, not a regression).

### Gaps

| ID | Severity | Area | Description |
|---|---|---|---|
| **G1** | **Medium** | Offline writes | The legacy app had a Map-based offline write queue in `FirestoreService` (flushed on the `online` event with success/failure toasts). The SvelteKit app uses TanStack Query's `networkMode: offlineFirst` for *reads* only — there is no client-side write outbox. Phase 8 tracker did not call this out. **Recommend** tracking as Phase 9 work; a Dexie-backed outbox was the original Phase 6 plan in `MIGRATION_PLAN.md`. |
| **G2** | Low | Notification typing | `notifications.type` is a free-form `text` column. Legacy used a fixed set; current code uses the same fixed set (`transaction_summary`, `transaction_upcoming`, `transaction_overdue`, `transaction_reminder`, `group_invitation`, `system_notification`) but nothing enforces it. **Recommend** a Postgres `notification_type` enum migration. |
| **G3** | Low | `notifications.data` shape | Per-type payload in `data jsonb` is undocumented. Each type has its own contract (e.g. `group_invitation` carries `groupId`, `inviterEmail`). **Recommend** a TypeScript discriminated union plus a row-level CHECK or `pg_jsonschema`. |
| **G4** | **Medium** | Vault secret rotation | `internal_trigger_secret`, `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT` live in Supabase Edge Function secrets and Vault. There is no documented runbook for rotation and no expiry. **Recommend** a `docs/runbooks/secret-rotation.md`. |
| **G5** | **Medium** | RLS regression tests | Phase 8 backlog mentions Vitest RLS tests using two `supabase-js` clients with different JWTs; none exist. RLS was verified manually in Supabase Studio only. **Recommend** writing them before further policy changes. |
| **G6** | Low | pg_cron DST drift | Schedules are UTC-anchored. Local fire time in Warsaw shifts ±1h at DST transitions twice a year. Acknowledged in `MIGRATION_PLAN.md`. **Recommend** explicitly documenting expected fire windows in the cron schedule comment. |
| **G7** | Low | Edge Function `deno.json` | All three Edge Functions are running but neither has a committed `deno.json` declaring imports / compiler options. Currently they import via JSR/npm specifiers inline. **Recommend** adding `deno.json` to lock versions. |

## Quality findings (from `get_advisors`)

### Security

| Lint | Detail | Action |
|---|---|---|
| `function_search_path_mutable` (multiple) | SECURITY DEFINER functions without `SET search_path = public` (e.g. `handle_updated_at`, `bump_last_used_at`, `mark_all_notifications_read`, etc.). Allows search-path-based privilege escalation if a same-named relation is created in a writable schema. | Add `SET search_path = public` (or `public, vault` where vault is needed) to every SECURITY DEFINER function. One migration. |

### Performance

| Lint | Detail | Action |
|---|---|---|
| `unindexed_foreign_keys` | `group_invitations.invited_user_id`, `shopping_lists.category_id`, `transactions.recurring_template_id`, `transactions.shopping_list_id` lack a covering index. | Add four FK-covering indexes. |
| `auth_rls_initplan` (×2) | `user_groups: members and owners can read` and `group_invitations: invited user can read by email` re-evaluate `auth.<function>()` per row. The `auth.uid()` calls *are* wrapped in `(select ...)`; the unwrapped call is `auth.jwt() ->> 'email'` (the `->>` operator). | Wrap as `(select auth.jwt() ->> 'email')`. |
| `unused_index` (multiple) | Several indexes (`idx_transactions_user_date_*`, `idx_transactions_recurring_template`, etc.) have never been used. Likely because the live database has very low row counts so far; `EXPLAIN ANALYZE` would just seq-scan. | Re-evaluate after the first month of real traffic; do not drop yet. |
| `multiple_permissive_policies` | Several tables (notably `categories`) have multiple permissive policies on the same `(role, action)` pair, each evaluated for every row. | Consolidate per-action policies where the predicates can be `OR`-ed without losing readability. Low priority — correctness first. |

### Operational

| Item | Detail | Action |
|---|---|---|
| **Migration drift** | `mcp__supabase__list_migrations` returns only the most recent (`20260504195407 admin_trigger_rpc`). All earlier migrations were applied before platform migration tracking was wired. The SQL files are the source of truth. | Re-import the missing entries into `supabase_migrations.schema_migrations` (idempotent fix), or accept the SQL files as canonical. Document either way. |
| **No backups runbook** | Supabase auto-backups are enabled at the platform level but the project has no documented restore procedure. | Add a `docs/runbooks/restore.md`. |
| **No rate limiting on RPCs** | `create_group`, `invite_user` lack rate limits. Personal app, low risk — but spammable in principle. | Defer; revisit if abuse occurs. |

## Architectural strengths (worth preserving)

- **Direct writes to `user_groups`, `group_members`, `group_invitations` are blocked**; all mutations go through SECURITY DEFINER RPCs. Clean privilege boundary that survives schema evolution.
- **RLS recursion was caught and fixed early** (migrations `20260424000000` / `20260424000001`) via two SECURITY DEFINER helpers. Subsequent policy work has not regressed.
- **All `auth.uid()` calls in RLS are wrapped** in `(select auth.uid())` — the standard initPlan optimisation that turns per-row evaluation into per-statement.
- **Hybrid scheduling** (pg_cron for SQL-only jobs, Edge Functions for HTTP-needing work) keeps the moving parts minimal. The single shared-secret indirection (`pg_net.http_post` + Vault) is the only auth model that needs to be understood.
- **VAPID web-push** drops the entire Firebase Messaging SDK (~200KB) and replaces it with native browser APIs.
- **Static SvelteKit + Cloudflare Pages** removes the SSR/server-runtime dimension entirely. The whole frontend is a static bundle.

## Recommendations — prioritised

1. **G5** (RLS regression tests) — write before any further RLS work. One Vitest spec per table is enough; two `supabase-js` clients with different JWTs.
2. **Security: function `search_path` pinning** — single migration, no behaviour change.
3. **Performance: four FK-covering indexes + two `auth.jwt()` wraps** — single migration.
4. **G4** (secret rotation runbook) — `docs/runbooks/secret-rotation.md`.
5. **G1** (offline write queue) — track as a Phase 9 item; Dexie-backed outbox per the original Phase 6 plan.
6. **G2 / G3** (notification type enum + payload schema) — combined migration.
7. **G7** (Edge Function `deno.json`) — three small commits, no behaviour change.
8. **G6** (cron DST documentation) — comment in the cron migrations.

None of these are migration blockers. The migration is **functionally complete**.

## Sign-off

The React+Firebase → SvelteKit+Supabase migration is feature-complete as of this audit. The items above are post-migration quality work; none of them gate declaring the migration shipped.
